# LLM-Assisted Community Signal Dashboard Scope

## Summary

Evolve the dashboard from raw metric monitoring into a DevRel-oriented signal dashboard. Keep `metrics`, `stats`, and `history` unchanged as source-of-truth tables. Add derived tables and jobs that precompute growth signals, collect first-class evidence, and generate weekly cached LLM guidance.

The v1 goal is to help DevRel answer: "What moved, what external evidence may explain it, and what action should we consider?" The dashboard should not call OpenAI or perform expensive rollups at request time.

## Key Changes

- Add derived LanceDB Enterprise tables:
  - `dashboard_metric_rollups`: precomputed 7d, 15d, 30d, 90d, last-full-month, prior-window, delta, percent-change, SDK share, and trend fields.
  - `evidence_items`: first-class external/manual evidence records.
  - `signal_candidates`: deterministic candidate signals generated from rollups and evidence.
  - `signal_guidance`: weekly cached LLM guidance generated from signal candidates, 7d/15d/30d rollups, and evidence.
- Keep source tables untouched:
  - `metrics`, `stats`, and `history` remain the only source-of-truth ingestion tables.
  - Derived jobs may read from source tables but must never alter source table schemas or semantics.
- Add a Hacker News evidence collector:
  - Use the public HN Algolia API.
  - Search terms: `LanceDB`, `lancedb`, `@lancedb/lancedb`, `lance format`, `lance file format`, `memory-lancedb`, `memory-lancedb-pro`, `OpenClaw lancedb`.
  - Store story/comment hits as `evidence_items`.
  - Deduplicate by HN object ID plus matched term.
  - Preserve a queryable HN occurrence date so the dashboard can later filter and display "most recent HN mentions."
- Defer Reddit automation:
  - For now, Reddit/F5bot findings can be manually entered as `evidence_items`.
  - Paid F5bot API/mailbox automation is out of scope for v1.
- Defer GitHub downstream dependency collection implementation:
  - Keep the design documented, but do not build in v1.
  - When built, only count exact dependencies found in package manifests or lockfiles, never README/free-text/code-comment matches.

## Interfaces And Data Model

- `evidence_items` should support:
  - `evidence_id`
  - `source_type`: `hackernews`, `manual`, later `github`, `reddit`, `npm`, `crates`
  - `source_name`
  - `observed_at`: when our collector/manual process recorded the evidence
  - `occurred_at`: when the external item was published/created; this is required and filterable for HN evidence
  - `title`
  - `url`
  - `snippet`: short display text for "most recent mentions" UI
  - `matched_terms`
  - `related_metrics`
  - `related_packages`
  - `related_repos`
  - `communities`
  - `evidence_strength`: `weak`, `medium`, `strong`
  - `raw_ref`
- For HN evidence:
  - `occurred_at` should map from the HN item creation timestamp.
  - `snippet` should use comment text/story text when available, otherwise title.
  - `url` should point to the HN item or external story URL consistently enough for dashboard display.
- `signal_candidates` should initially support:
  - `download_spike`
  - `sustained_growth`
  - `sdk_share_shift`
  - `social_mention_burst`
- Signal candidates are generated by deterministic rules:
  - `download_spike`: current window materially exceeds prior equivalent window.
  - `sustained_growth`: positive trend across multiple recent monthly buckets.
  - `sdk_share_shift`: one SDK's share changes materially within a product family.
  - `social_mention_burst`: enough HN/manual evidence items appear in a recent window.
- Dashboard API should read derived tables for rollups/signals/evidence instead of recomputing all dashboard values from raw `stats` rows on each request.
- LLM guidance is generated by `generate_signal_guidance` using OpenAI Responses API, defaults to `gpt-5.5` with high reasoning, and is stored in `signal_guidance`.
- The weekly LLM job uses the latest 7d rollups as the primary assessment window, compares against 15d and 30d rollups, includes 7d evidence details, and includes 15d/30d evidence summaries.
- Guidance must cite concrete `signal_id`, `rollup_id`, or `evidence_id` values and must not invent evidence, communities, repositories, packages, or causes.

## GitHub Downstream Collector Design For Later

- GitHub search should only find candidate manifest files.
- Dependency evidence is valid only after parsing manifests or lockfiles.
- Valid files:
  - Python: `pyproject.toml`, `requirements.txt`, `setup.py`, `setup.cfg`, `Pipfile`, `poetry.lock`, `uv.lock`
  - Node: `package.json`, `package-lock.json`, `pnpm-lock.yaml`, `yarn.lock`, `bun.lock`
  - Rust: `Cargo.toml`, `Cargo.lock`
- Exact package identities:
  - PyPI `lancedb` -> `downloads:lancedb:python`
  - PyPI `pylance` -> `downloads:lance:python`
  - npm `@lancedb/lancedb` -> `downloads:lancedb:nodejs`
  - crates `lancedb` -> `downloads:lancedb:rust`
  - crates `lance` -> `downloads:lance:rust`
- Do not count free-text mentions, README mentions, issue/comment mentions, or arbitrary `import lancedb` matches as downstream dependency evidence.

## Test Plan

- Unit test rollup calculations:
  - 7d/15d/30d/90d totals.
  - prior-window deltas and percent changes.
  - last-full-month totals.
  - SDK share calculations.
- Unit test signal generation:
  - spike detected when current window exceeds threshold.
  - sustained growth detected from monthly trend data.
  - SDK share shift detected across product metrics.
  - HN burst detected from evidence count in window.
- Unit test HN evidence normalization:
  - search result maps to `evidence_items`.
  - HN creation timestamp maps to filterable `occurred_at`.
  - display snippet is populated from text/title.
  - dedupe key prevents duplicate insertions.
  - matched terms and related metrics are populated consistently.
- Regression test source-table safety:
  - derived jobs do not mutate `metrics`, `stats`, or `history`.
- Frontend/API test:
  - dashboard can render from derived rollup/signal/evidence/guidance payloads without requiring raw source-row materialization.
  - dashboard can request recent HN evidence ordered by `occurred_at`.

## Assumptions

- LanceDB Enterprise remains the only storage target.
- Existing source tables are not changed.
- Derived jobs are allowed to upsert/delete/recompute derived tables.
- HN is the only automated external evidence source in v1.
- OpenAI is used only by the weekly cached guidance job, not by dashboard requests.
- Reddit/F5bot evidence starts as manual entry.
- GitHub downstream dependency tracking is important, but implementation is deferred until after the derived-table and HN evidence pattern is working.
